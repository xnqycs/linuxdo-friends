# linux.do 好友增强浏览器插件：必要性与可行性分析

生成时间：2026-06-27

## 结论

需求本身成立，但产品边界必须收窄。

值得做的不是“监控好友在线状态”的工具，而是“在用户自己的浏览器里增强 linux.do 好友/关注管理”的工具：用户手动维护好友名单、分组、备注和关注理由；插件低频读取用户登录后可见的公开页面/API，汇总好友近期发帖、回帖、最后发帖时间等活动线索；尽量避免实时在线追踪、批量扫站和隐私化画像。

推荐的 MVP 是：

1. 好友名单增强：基于 Discourse 原有关注系统导入；只有已关注用户才能升级为“特别关注”。
2. 好友备注和分组：本地存储，不上传。
3. 活动摘要：最近发帖/回帖/主题，按低频刷新和用户主动刷新获取。
4. 页面增强：在用户卡片、帖子楼层、个人页上标记“好友/分组/备注”。
5. 风控保护：默认不后台高频轮询，不抓取非用户可见内容，不绕过 Cloudflare。

## 必要性分析

### 真实痛点

linux.do 当前可用的社交关系主要是关注。关注适合表达“我想看到这个人”，但不适合承载完整好友管理：

- 关注关系没有强分组语义，难以区分朋友、熟人、技术同好、交易对象、项目协作者。
- 关注流通常偏内容消费，不偏关系维护。用户想知道“这个朋友最近在不在、最近活跃在哪些话题”，需要反复点个人页。
- 社区里弱关系多，用户需要备注上下文，例如“上次聊过 NAS”“帮我测试过脚本”“可信交易对象”。
- linux.do 讨论密度高时，好友的新动态容易被全站信息流淹没。

这些都说明“好友增强”有必要。

### 需要避免的错误方向

“跟踪最近在线”这个表述有风险。它容易滑向被动监控，尤其是：

- 高频检查某用户是否刚刚在线。
- 生成精确在线/离线时间线。
- 对别人不可见或难以察觉的行为做聚合画像。
- 后台绕过站点风控持续请求。

这类方向不但容易引起用户反感，也更可能触发站点风控或违反社区预期。

更稳的定位是“好友活动摘要”，不是“在线监控”。

## 可行性分析

### 数据层可行

linux.do 基于 Discourse 形态，Discourse 官方 API 文档显示存在用户资料、用户行为、搜索、目录等端点，包括：

- `/u/{username}.json`：单个用户资料，官方 schema 包含 `last_posted_at`、`last_seen_at` 等字段。
- `/user_actions.json`：按用户名和类型读取用户行为列表。
- `/search.json`：可用 `@username` 等查询语法查找用户内容。
- `/directory_items.json`：用户目录数据。

因此，活动摘要在技术上有路可走。但字段是否在 linux.do 对普通登录用户开放、是否隐藏、是否被权限配置改写，需要在真实登录浏览器环境里验证。

### 访问层有明显风控约束

本地命令直接访问 `https://linux.do/u/neil.json` 时，返回 Cloudflare challenge；连续访问后出现 `HTTP/2 429` 且响应头包含 `cf-mitigated: challenge`。这说明插件不能设计为后台脚本或外部服务高频抓接口。

合理方式是：

- 只在用户已登录 linux.do 的浏览器上下文内工作。
- 优先由 content script 增强当前页面。
- 后台刷新必须低频、可关闭、有退避策略。
- 可以在后台静默打开非激活标签页读取用户登录态下可见页面，但只能作为低频、可解释、可关闭的降级路径；打开、读取、关闭都要有超时和退避。
- 不绕过 Cloudflare，不伪装自动化请求，不做服务端集中采集。

### 浏览器插件层可行

Chrome Manifest V3 支持完成这类工具的基本能力：

- content script：读取和增强 linux.do 页面。
- host permissions：限定只访问 `https://linux.do/*`。
- storage：保存好友名单、分组、备注、最近摘要和设置。
- alarms：做低频周期刷新，但不能依赖长期常驻后台。
- tabs：在需要登录态页面渲染时，可由 service worker 或 extension page 创建 `active:false` 的后台标签页辅助取数；content script 不直接调用 tabs API。
- action popup / side panel：展示好友活动面板。

技术栈上，React + Jotai 适合第一版：

- React 负责 popup、side panel、options 页面和页面内注入面板。
- Jotai 用原子状态拆分关注列表、特别关注列表、活动缓存、同步状态、设置项和错误状态。
- service worker 只做调度、标签页任务、消息转发和存储读写，不承载复杂 UI 状态。
- content script 负责从当前 linux.do 页面提取可见用户信息、插入标记、接收 popup/worker 消息。

这类插件不需要远程后端。第一版应坚持本地优先，避免账号、隐私和部署复杂度。

## 可用技能与工具

当前 Codex 环境里没有找到专门的“浏览器插件制作” skill：

- 已安装 skill 中没有 Chrome extension / browser extension / React 专用制作 skill。
- 官方 curated skill 列表未筛到浏览器插件制作项。
- 官方 experimental skills 路径当前不可用。

后续开发可用的相关能力是：

- `chrome:control-chrome`：用于真实 Chrome 登录态验证、加载 unpacked extension、检查扩展行为。
- `browser:control-in-app-browser`：用于普通本地页面/开发服务验证。
- `skill-installer`：后续如果发现第三方或官方新增浏览器插件 skill，可安装。

因此第一版实现建议按普通前端工程搭建，而不是依赖专用 skill。

## 推荐产品边界

### 第一版应该做

- 同步 Discourse 关注列表。
- 只允许把已关注用户加入“特别关注”。
- 对特别关注用户做本地备注、标签、分组、置顶。
- 好友活动摘要：最近主题、最近回复、最后发帖时间。
- 在帖子列表/帖子详情/用户卡片显示好友标记和备注。
- 用户主动刷新；可选低频自动刷新，例如每 30 到 120 分钟。
- 必要时使用后台非激活标签页读取登录态页面，但要限制并发、频率、生命周期和失败重试。
- 请求失败、被 challenge、被限流时清晰降级，不重试轰炸。

### 第一版不应该做

- 实时在线监控。
- 精确在线时间线。
- 后台全天候高频轮询。
- 采集非好友的大范围用户目录。
- 把未关注用户加入特别关注。
- 上传好友关系或活动数据到第三方服务。
- 绕过 Cloudflare、验证码或站点权限。

## 命名

项目命名确定为：

- 仓库名：`linuxdo-friends`
- 中文展示名：`佬朋友`
- 英文展示名：`LinuxDo Friends`
- Manifest 展示名：`佬朋友 - LinuxDo Friends`
- 核心动作：`升级为好友`
- 核心功能名：`特别关注`

命名逻辑：

- `佬` 保留 linux.do 社区语境，但不用 `佬友` 这个已有社区称呼。
- `朋友 / Friends` 直接对应“从关注升级为朋友”的产品概念。
- `特别关注` 留在 UI 功能层，表达从已关注用户里挑出需要重点看的朋友。
- 避免 `Tracker`、`Monitor`、`Spy`、`Online` 等跟踪感词汇。

## 不对抗 Cloudflare 的取数方式

静默标签页不是唯一方案，也不应该是默认方案。推荐顺序是：

1. **用户主动刷新**

   在 popup 或 side panel 里提供“同步关注列表”“刷新特别关注近况”按钮。用户明确触发后，再从当前登录态读取数据。优点是透明、容易解释；缺点是自动化程度低。

2. **已有 linux.do 标签页内取数**

   如果用户已经打开 linux.do 标签页，插件向该标签页的 content script 发消息，让它在同源页面环境里请求或解析登录态可见数据。优点是不用额外开标签页，也更接近正常页面行为；缺点是要求至少有一个可用的 linux.do 页面。

3. **扩展后台直接 fetch JSON**

   service worker 在具备 `host_permissions` 的前提下尝试请求 Discourse JSON 端点，并显式带上用户登录态所需的凭据。优点是简单；缺点是它可能不像真实页面环境，遇到 Cloudflare challenge 或 403/429 时必须立即退避，不做重试轰炸。

4. **后台非激活标签页**

   只有当前面方式不稳定时才用：创建 `active:false` 标签页，加载登录态页面，等待正常页面完成后读取 DOM 或 JSON，再关闭。它不破解 challenge，只是让浏览器按用户会话正常打开页面。必须限制并发、频率、生命周期、失败退避，并在设置里说明。

5. **用户导入/手动维护**

   允许用户从当前关注页导入、手动添加已关注用户名、手动标记特别关注。它不是实时方案，但能保证核心关系管理不依赖后台抓取。

暂不推荐“当前页面机会式采集”作为第一版核心逻辑。它虽然不额外制造请求，但数据来源不稳定、状态一致性差，也容易让用户困惑“为什么有些人自动更新、有些人不更新”。第一版更适合把刷新行为做成显式动作。

明确不做：

- 不解 Cloudflare challenge。
- 不模拟验证码、不接码、不自动过人机校验。
- 不伪造浏览器指纹或绕过风控头。
- 不读取或导出浏览器 cookie。
- 不用远程服务器代请求。
- 不在 403/429/challenge 后持续重试。

## 风险与缓解

### 隐私风险

风险：把“朋友近况”做成“他什么时候在线”的监控。

缓解：默认只展示公开活动摘要；对 `last_seen_at` 这类字段谨慎使用，最多显示粗粒度状态，例如“近期活跃”，不要显示精确在线轨迹。

### 风控风险

风险：批量请求用户 JSON 或 user_actions 触发 Cloudflare / 429。

缓解：用户主动刷新优先；自动刷新低频；按好友数量分批；指数退避；缓存结果；页面可见时才刷新。

### 兼容风险

风险：linux.do 页面结构或 Discourse 配置变化导致选择器/API 不稳定。

缓解：优先使用 Discourse JSON API；页面增强选择器集中封装；失败时只关闭增强层，不影响原站使用。

### 社区接受度风险

风险：工具被理解成跟踪器。

缓解：命名、文案、默认功能都避开“监控在线”；强调本地、低频、可见数据、用户自用。

## 建议的下一步

1. 在已登录浏览器里验证 3 个端点：`/u/{username}.json`、`/user_actions.json`、`/search.json?q=@username`。
2. 验证关注列表来源和已关注判定方式，确定“特别关注”只能来自已关注集合。
3. 确认 linux.do 普通用户是否能看到好友目标的 `last_seen_at`，以及该字段是否对隐私设置敏感。
4. 做一个无后端 MV3 原型：关注同步、特别关注、本地备注、当前页好友标记、手动刷新最近活动。
5. 用真实 20 到 50 个特别关注样本测试请求频率、后台标签页策略和页面稳定性。

## 证据

- Discourse API 文档说明大多数 HTML 端点有 JSON 对应形式，并列出 `/u/{username}.json`、`/user_actions.json`、`/search.json`、`/directory_items.json`。
- Discourse `/u/{username}.json` schema 包含 `last_posted_at`、`last_seen_at` 等用户资料字段。
- 本地访问 `https://linux.do/u/neil.json` 收到 Cloudflare challenge，随后出现 `HTTP/2 429` 与 `cf-mitigated: challenge`。
- Chrome 扩展官方文档提供 MV3 权限、storage、alarms、tabs 等能力，适合本地优先的页面增强插件。
- Jotai 官方文档说明其以 atom 方式组织 React 全局状态，适合拆分插件 UI 的小状态单元。
